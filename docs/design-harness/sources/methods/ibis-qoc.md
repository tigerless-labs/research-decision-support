# [Method, cautionary tale] IBIS / QOC -- the design-rationale family killed by capture cost

IBIS (Rittel, 1970, designed for wicked problems). Tooled as gIBIS/Compendium. QOC (MacLean et
al., 1991, HCI design rationale).

Core logic: structure design discussion into a multi-noun ontology -- IBIS uses
issue/position/argument, QOC uses question/options/criteria. Academically it validated that
"structuring rationale has value"; in engineering the whole family died. The autopsy conclusion
(Grudin's paradox): **the person bearing the recording cost is not the one who benefits** --
engineers found filling in the structure too tedious, the benefit went to future readers, and
capture cost killed the entire category.

Boundaries: as a structural ontology it remains valid (QOC's question + options is exactly the
decision card's skeleton); what died was "humans filling in forms" as the execution mode.

**Relation to this project**: the direct basis for two core decisions -- keep only one human-facing
entity (more nouns = more form-filling obligations), and hand all bookkeeping labor to the agent.
The product's entire why-now hypothesis: the agent drives capture cost to zero, so IBIS's cause of
death no longer holds today (hypothesis pending prototype validation, see the decisions layer).
