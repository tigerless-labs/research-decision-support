# [GitHub] tinyhumansai/OpenHuman

[github.com/tinyhumansai/openhuman](https://github.com/tinyhumansai/openhuman) · Rust+Tauri desktop personal-AI agent; GitHub-Trending May 2026. GPL-3.0, early beta. Blogs: [mager.co explainer](https://www.mager.co/blog/2026-05-25-openhuman-explainer/).

The **Karpathy LLM-wiki pattern, productized**. Memory is a **Memory Tree**: every connected source (118+ OAuth integrations, auto-fetched every 20 min) is canonicalized → **≤3k-token Markdown chunks → scored → folded into hierarchical summary trees**, stored in local **SQLite** and dual-written as an editable **Obsidian `.md` vault**. Explicitly inspired by Karpathy's "LLM Knowledgebase / obsidian-wiki" (Apr-2026 gist): compile knowledge once into structured interlinked markdown and read the compiled output, instead of RAG re-deriving per query. `TokenJuice` compresses tool/scrape output (HTML→MD, dedup) before any LLM call (claims −80% tokens).

**File vs vector — file-based by construction; query-time recall undocumented.** The build side (chunk→score→fold→SQLite+vault) is well-documented; the **retrieval algorithm at query time is NOT stated** in the README or explainer (tree-traversal vs keyword vs vector — unspecified), and the project makes **no explicit anti-embedding claim** (that framing is third-party review, not the project). The Karpathy pattern is non-RAG *in spirit* (navigate a maintained wiki/tree, not vector-retrieve), so the architecture implies hierarchical navigation, but the algorithm is unverified here — would need a source check (Rust core). Contrast peers: [Hermes](nousresearch-hermes-agent.md) self-learns from your activity; [OpenClaw](openclaw.md) is BYO-memory; OpenHuman pre-wires integrations and the tree "builds itself." **Skills:** OpenHuman ships a `skill_registry` (`src/openhuman/skill_registry/`) that installs from the **HermesHub catalog** (`skills.json` — it consumes the Hermes ecosystem) plus a `create_skill` delegate sub-agent (`agent_registry/agents/skill_creator/`); but its central learning loop is **external-source → memory tree** (`SourceKind` = Chat/Email/Document from Gmail/Notion/Drive), **not** distillation of the agent's own task trajectory the way Hermes does.

**Relevance to autoharness.** Strong thesis-alignment, different artifact. Karpathy's LLM-wiki = **"maintain a structured, human-inspectable text store; compile once, don't re-derive"** — the same anti-RAG, structural-text stance behind our spine (structural-first, no oracle) and our human-adoption/inspectability bias. Its **hierarchical summary tree** is a memory structure to contrast with our **typed skill DAG**: both are *navigable structured text*, not embeddings, but a tree encodes summarization depth while our DAG encodes typed relations (dependency/conflict/dup). Open question worth tracking: how a Karpathy-tree does query-time selection without vectors — directly informs our Retrieve layer's "structure over similarity" positioning (see [synthesis/](../../synthesis/index.md)).
