# [GitHub] NousResearch/Hermes-Agent

[github.com/NousResearch/Hermes-Agent](https://github.com/NousResearch/Hermes-Agent) · open-source agent with memory + self-learning.

Autonomous **skill creation after complex tasks**; skills self-improve in use; periodic memory nudges (interval 10); FTS5 session search + LLM summarization; a `memory` tool (add/replace/remove) with capacity-driven consolidation; [Honcho](plastic-labs-honcho.md) user modeling. Also feeds back to the model level: batch trajectory generation + trajectory compression to train next-gen tool-calling models.

**Recall is pure-file / keyword — no vector retrieval (by default).** Skills under `~/.hermes/skills/<cat>/<name>/SKILL.md` are selected by **3-level progressive disclosure**: Level-0 descriptor index (name+description) in the prompt → `skill_view(name)` loads the body on demand → `skill_view(name, path)` for a reference; the LLM picks by reading descriptors (no similarity step). Memory = two always-injected flat files (`MEMORY.md` ~2200 char, `USER.md` ~1375 char, frozen snapshot for prefix-cache); past-chat recall = **FTS5 keyword** search over a SQLite SessionDB (docs: *"Keyword only, no vector/semantic"*). Vector/embeddings exist only as **open proposals** (issues #346/#531/#844/#10355) and **optional external plugins** (Honcho/Mem0/Chroma-skill), never the default path.

**Three production/maintenance paths — different trigger, different agent.** (1) *Autonomous creation* is gated by a **nudge counter**: `_iters_since_skill` accumulates one per tool-calling iteration (**across turns**) and, at turn finalization, once it ≥ `_skill_nudge_interval` (default **10**, config `skills.creation_nudge_interval`) AND `skill_manage` is available, it resets and calls `_spawn_background_review`. That **forks a daemon-thread AIAgent** *after the response is already delivered* (never competes for the live turn): same main model, **inherits the parent's cached system prompt → hits the same warm prefix cache**, whitelisted to memory+skill tools only, **replays the just-finished transcript**, and writes SKILL.md / memory straight to disk. The main conversation + its cache are never touched. Memory is symmetric (`_memory_nudge_interval` default 10). (2) *User-triggered* `/learn` (`build_learn_prompt`) is the **inline** path: the live agent authors the SKILL.md via `skill_manage` as a normal turn — no fork, no separate distillation engine. (3) *Consolidation* is a **separate forked aux-model curator** (`maybe_run_curator`): idle-triggered (≥ `min_idle_hours` 2h AND last run > `interval_hours` 7d), umbrella-building / pin / archive / patch; only touches `created_by: agent` skills, **never auto-deletes, only archives**; **LLM consolidation OFF by default** (`DEFAULT_CONSOLIDATE = False`), default is just the deterministic inactivity state machine active→stale(30d)→archived(90d). **State machine anchors on `last_used_at`** (`created_at` fallback if never used). **`stale` has zero behavioral effect — pure bookkeeping marker**: `set_state(STALE)` only writes a flag to the usage sidecar; the dir never moves, and repo-wide nothing in the recall / descriptor-index / loader path reads `state==stale` (only the curator's own next tick does), so a stale skill is recalled and usable exactly like active. **`archived` is the only state that changes recall, and it does so by physically `rename`-ing the dir into `.archive/`** (out of the skills tree → descriptor index can't see it), not by reading the flag; recoverable via `restore_skill`. Used again before 90d while stale → reactivate to active.

**Relevance to autoharness:** the shipped reference for separating **production** from **consolidation** — and for keeping production cheap. The maintain-not-grow *production sample* is distilled by a **background fork that replays the just-finished trajectory while the parent's prefix cache is still warm** — i.e. at the cheapest possible moment to re-read the trace — and runs **off the user's critical path**; the user can also trigger it inline via `/learn`. Only **consolidation** pays for a separate aux-model agent, and even that conservatively archives (never deletes) and is opt-in. add/replace/remove + capacity-driven consolidation = maintain-not-grow in production.
